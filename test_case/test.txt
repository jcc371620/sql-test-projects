1. 测试用例设计：多抓鱼“扫码回收”功能
题目：请为扫码回收图书功能设计测试用例。
解答（按维度拆分）：

功能性 (Functional)：
扫描标准 ISBN 码，是否能正确识别书名、作者、原价。
扫描非图书类二维码（如微信名片），系统是否报错提示“非图书条码”。

兼容性 (Compatibility)：
在光线昏暗、条形码部分污损、反光的情况下，识别率如何。
不同型号手机（iOS/Android）的摄像头对焦和调取权限是否正常。

异常场景 (Negative/Exception)：
网络中断：扫码过程中断网，提示是否友好，恢复后能否继续。
重复录入：同一本书扫两次，系统是增加数量还是提示“已在清单中”。
非法状态：账号被封禁或未登录时，点击扫码按钮的响应。

2.技术基础题：API 与 SQL
题目 A：HTTP 401 和 403 错误有什么区别？
401 Unauthorized：你是谁？（身份未验证）。比如你没传 API Key 或者 Token 传错了。
403 Forbidden：我知道你是谁，但你没权限。（权限不足）。比如普通用户尝试访问后台管理接口。

3.场景题：高并发下的库存扣减
题目：如果一本书只有 1 本库存，两个用户同时点击“购买”，如何测试？
解答：
测试点：是否存在“超卖”现象（即 1 本书卖给了 2 个人）。
测试方法：
使用工具（如 JMeter 或 Python 的 threading 库）模拟两个并发请求。
预期结果：其中一个请求成功（返回 200），另一个请求失败（返回“库存不足”或“手慢了”）。
底层原理：检查后端是否使用了 Database Lock (数据库锁) 或 Redis Distributed Lock (分布式锁)。

4.AI 测试：Prompt 设计
题目：如何利用 AI 提高测试效率？
解答：
使用 Prompt Engineering 生成测试数据。
Bad Prompt: "帮我写测试数据。"
Good Prompt (CO-STAR 框架):

Context: 我在测试多抓鱼的图书回收接口。
Objective: 生成 10 条 JSON 格式的图书数据。
Style: 包含多种 ISBN 格式、长书名、特殊字符作者名。
Response: 请直接输出 JSON 数组。

5. 回收履约流转测试
题目：多抓鱼的回收流程是“用户下单 -> 顺丰上门收书 -> 仓库签收 -> 审核鉴定 -> 自动放款”。请针对“审核鉴定”到“自动放款”这一环节，设计 3 个关键的测试点。
解答思路：

逻辑判断测试：如果书籍鉴定结果为“拒绝回收”，系统是否能正确拦截放款流程？
金额准确性测试：如果书籍有破损需要扣款，最终放款金额是否等于 算法估价 - 破损扣减？
状态机测试：书籍进入仓库后，订单状态是否从“待签收”准确变为“鉴定中”，鉴定完成后是否实时触发“已打款”通知？

6.如何利用 AI 改进多抓鱼的测试？
题目：多抓鱼每天有大量非标品（旧衣服、旧书）入库，鉴定标准复杂。作为测试，你如何利用 AI 或自动化工具提升效率？

解答思路：
图像识别测试：引入 AI 视觉模型辅助鉴定书籍破损程度，测试时对比 AI 鉴定结果与人工结果的重合度。
自动化脚本：针对回收下单的 Regression Testing (回归测试)，编写 Python + Playwright 脚本，确保每次系统更新后，用户依然能正常下单。
数据构造：利用 Prompt Engineering (提示工程) 快速生成各种极端的书籍 metadata（元数据）来填充数据库，测试系统的压力承载。
